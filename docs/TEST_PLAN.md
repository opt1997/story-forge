# Story Forge 基础测试用例

更新时间：2026-07-04

本文是可执行的手动测试清单，不包含自动化测试代码。每次较大改动后，应至少跑完基础路径。

## 0. 测试准备

1. 确认依赖已安装。
2. 启动本地服务。
3. 打开 `http://localhost:3000`。
4. 如果使用真实 DeepSeek/OpenAI，确认环境变量已在本机设置，但不要写入仓库。
5. 如需测试作品库，准备若干 `stories/{story_id}/` 文件夹，每个有效作品至少包含 `story.meta.json` 和 `final.md`。

## 1. 推荐主题测试

1. 打开首页，应该展示 5 个推荐主题。
2. 后台候选主题总数应为 30 个。
3. 点击“刷新主题”，应重新生成 30 个候选主题。
4. 刷新页面后，推荐主题仍然可读取。
5. 点击“用此主题生成”，应创建一个任务。
6. 推荐主题卡片应包含标题、摘要或核心说明、题材/标签、热度或质量相关分数。
7. 刷新主题不应删除已有任务。

## 2. 当前任务测试

1. 点击“开始创作”，应创建新任务。
2. 重复点击，应创建多个任务，不覆盖旧任务。
3. 当前任务只展示最新 5 条。
4. 最新任务应显示在最上方。
5. 点击单条任务删除按钮，该任务应从当前任务中消失。
6. 点击“一键清空”，当前任务列表应清空。
7. 被删除或清空的任务不应刷新页面后重新出现。
8. Writer / QA / Rewrite 轮次应正确显示。
9. 选中任务后，右侧栏应切换为任务详情。
10. 任务详情应展示 task_id、story_id、当前阶段、当前状态、创建/更新时间和 pipeline 信息。

## 3. 作品库测试

1. 在 `stories/` 下存在有效作品文件夹时，作品库应展示该作品。
2. 没有 `story.meta.json` 的文件夹不应展示。
3. 没有 `final.md` 的文件夹不应展示。
4. `story.meta.json.type !== "story"` 的文件夹不应展示。
5. `status !== "done"` 的文件夹不应展示。
6. 作品库每页只展示 10 篇。
7. 作品库按 `created_at` 倒序排列。
8. 修改 `read_count` 并保存后，`story.meta.json` 应更新。
9. 修改 `drop_off_users` 并保存后，`story.meta.json` 应更新。
10. 选中作品后，右侧栏应切换为作品详情。
11. 作品详情应展示标题、摘要、标签、阅读量、触底人数和正文预览。
12. 点击作品库头部“打开文件夹”，应打开本地 `stories/` 文件夹。
13. 点击单篇作品的“打开”，应打开该作品文件夹。

## 4. 页面结构测试

1. 页面不应展示“今日创作策略”模块。
2. 页面右上角不应展示“刷新”按钮。
3. “作品库与指标回填”应改名为“作品库”。
4. 右侧栏不应展示大量 missing 文件状态。
5. 左侧导航文案应符合中文用户习惯。
6. 点击左侧导航时，选中项底色应变化。
7. 点击左侧导航时，中间内容应滚动到对应模块。
8. 页面下滑时，左侧导航应保持可见。
9. 顶部主按钮文案应为“开始创作”。
10. 作品库标题和文件夹 ID 应在同一行，以标题为主、ID 为辅。

## 5. story.meta.json 测试

1. 有效 `story.meta.json` 应能被作品库读取。
2. `type` 缺失或不是 `"story"` 时，该作品不进入作品库。
3. `status` 缺失或不是 `"done"` 时，该作品不进入作品库。
4. `read_count` 缺失时，前端应按 0 处理。
5. `drop_off_users` 缺失时，前端应按 0 处理。
6. 保存指标后，`updated_at` 应更新。
7. 保存指标后，刷新页面应显示保存后的值。

## 6. Provider 与运行模式测试

1. 未设置真实 provider 时，系统应以 mock 模式运行。
2. 设置 `STORY_FORGE_LLM_PROVIDER=deepseek` 且存在 `DEEPSEEK_API_KEY` 时，runtime 应显示 API 模式。
3. 设置 `STORY_FORGE_LLM_PROVIDER=deepseek` 但缺少 `DEEPSEEK_API_KEY` 时，应显示明确的缺 key 状态或错误。
4. 不应把 API key 展示在页面正文中。
5. 不应把 API key 写入任何 tracked 文件。

## 7. 基础构建测试

1. 运行 `npm run build`。
2. 构建应通过。
3. 如果出现当前已知 Turbopack NFT tracing warning，只要编译成功即可视为非阻塞。
4. 构建不应生成需要提交的运行缓存、日志或密钥文件。

## 8. 回归检查

1. 不应新增业务 Agent。
2. 不应重构 Workflow 顺序。
3. 不应把作品库数据源改成纯数据库。
4. 不应删除本地 `stories/` 作品文件。
5. 不应接入番茄后台。
6. 不应加入自动发布逻辑。
